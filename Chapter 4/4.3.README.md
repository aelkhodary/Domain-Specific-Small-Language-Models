# 4.3 Areas for Improvement (Cost Savings and Performance)

## Simple Summary

This section explains how to **get the most out of your GPU** when running LLMs and identify **bottlenecks** that slow things down.

---

## 4.3.1 Getting the Most from Your GPU

### The Three Key GPU Specs

| Spec | What It Means | T4 GPU Example |
|------|---------------|----------------|
| **Compute Throughput** | How fast the GPU can do math | 65 TFLOPS |
| **Memory** | How much data the GPU can hold | 16 GB |
| **Memory Bandwidth** | How fast data moves to/from memory | 300 GB/sec |

### The Big Question: What's Slowing You Down?

Your GPU can either be:
- **Memory-bound** → Data can't move fast enough (GPU is waiting for data)
- **Compute-bound** → GPU can't calculate fast enough (data is waiting for GPU)

**How to tell which one:**
```
ops/byte ratio = Compute Speed / Memory Bandwidth
```
For T4 GPU: `65,000 GFLOPS / 300 GB/sec = 216.6 ops/byte`

This means the GPU can do **216.6 calculations** for every byte it reads. If your model needs fewer ops per byte, you're **memory-bound** (the common case for LLMs).

---

## The KV Cache: A Must-Have Optimization

### What Is It?

The **Key-Value (KV) Cache** stores attention computation results so you don't recalculate them for every new token.

### How LLM Generation Works (Two Phases)

| Phase | What Happens | KV Cache Role |
|-------|--------------|---------------|
| **Prefill** | Process all prompt tokens at once | Fills the cache with initial state |
| **Autoregressive** | Generate tokens one-by-one | Reuses cached values instead of recomputing |

### Why It Matters (Real Numbers)

```
With KV Cache:    10.3 seconds (average)
Without KV Cache: 52.0 seconds (average)
```

**Result: 5x faster with KV cache enabled!**

### Key Takeaway
Always use `use_cache=True` (default in Transformers). The small extra memory cost is worth the massive speed boost.

## Understanding Arithmetic Intensity

### What Is It?

**Arithmetic Intensity** = Total Compute Operations / Total Memory Movement

It tells you how much "work" the GPU does per byte of data moved.

### GPT-Neo Example

| Metric | Value |
|--------|-------|
| Total Memory Movement | 71.3 MB |
| Total Compute | 2.16 billion ops |
| **Arithmetic Intensity** | **30.29 ops/byte** |

### The Verdict

- GPU can handle: **216.6 ops/byte**
- Model needs: **30.29 ops/byte**
- **Gap: GPU is 7x more powerful than needed!**

This means the system is **memory-bound** — the GPU sits idle waiting for data. The next section covers how to fix this.

---

## Quick Reference: Attention Formula

The attention mechanism works in 3 steps:

```
Step 1: S = Q × K^T        (Query times Key-transpose)
Step 2: P = softmax(S)     (Normalize the scores)
Step 3: O = P × V          (Apply to Values)
```

Each step involves: **Load → Compute → Store**

This constant data movement is why LLMs are often memory-bound.

---

## Key Takeaways

1. **Know your bottleneck** — Most LLM inference is memory-bound, not compute-bound
2. **Always use KV cache** — 5x speedup for minimal memory cost
3. **ops/byte ratio** — Helps you understand if you're using your GPU efficiently
4. **Arithmetic intensity** — Lower values = memory-bound, higher values = compute-bound
5. **The 20% overhead** in memory formulas accounts for things like KV cache storage
