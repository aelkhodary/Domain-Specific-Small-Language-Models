# Chapter 4: Calculating Inference Cost

## 4.2 Calculating Inference Cost

Deploying an LLM to production to handle multiple inference requests costs money. In the previous section, you learned that larger models cost more because their weights require more GPU memory, and you often need extra headroom to maintain performance. You also saw a CLI tool in the Accelerate library that estimates the required memory. It's worth understanding how it makes that calculation so you can estimate memory needs when serving LLMs outside the Transformers library and make other sizing decisions.

Model documentation for available pretrained open source models usually lists the number of trainable parameters, but for deployment, you need the model's in-memory size so you can decide which GPU (or combination of GPUs) you'll need.

### The Formula

**Memory (GB) = 4 × P × (F / 32) × 1.2**

Where:
- **4** = the number of bytes for each trainable parameter
- **P** = the model's total number of trainable parameters (in billions)
- **F** = the number of bits for the chosen precision (32 for FP32, 16 for FP16, 8 for INT8)
- **1.2** = a factor that includes an average 20% overhead due to several factors (detailed later in this chapter)

### Practical Examples

| Model | Parameters | Precision | Memory Calculation | Result |
|-------|------------|-----------|-------------------|--------|
| GPT-Neo | 2.7B | FP32 | 4 × 2.7 × 1 × 1.2 | ~12.96 GB |
| GPT-Neo | 2.7B | FP16 | 4 × 2.7 × 0.5 × 1.2 | ~6.48 GB |
| Mistral | 7B | FP16 | 4 × 7 × 0.5 × 1.2 | ~16.8 GB |
| Mistral | 7B | INT8 | 4 × 7 × 0.25 × 1.2 | ~8.4 GB |

## Understanding Precision (F) in LLM Inference

**Precision** refers to how many bits are used to represent each number (weight) in the model. This is a trade-off between **accuracy** and **memory/speed**.

### The Three Common Formats

| Format | Bits | Range | Use Case |
|--------|------|-------|----------|
| **FP32** (Float32) | 32 bits | ±3.4 × 10³⁸ | Full precision, training default |
| **FP16** (Float16) | 16 bits | ±65,504 | Inference, mixed-precision training |
| **INT8** (Integer8) | 8 bits | -128 to 127 | Quantized inference, edge deployment |

### How Each Format Works

#### FP32 (32-bit Floating Point)
- Uses 1 sign bit, 8 exponent bits, 23 mantissa bits
- Highest precision, largest memory footprint
- Standard for training because gradient updates need precision

```
Structure: [S][EEEEEEEE][MMMMMMMMMMMMMMMMMMMMMMM] → 32 bits per weight
```

#### FP16 (16-bit Floating Point)
- Uses 1 sign bit, 5 exponent bits, 10 mantissa bits
- Half the memory of FP32 with minimal quality loss for inference
- Modern GPUs (NVIDIA Tensor Cores) are optimized for FP16

```
Structure: [S][EEEEE][MMMMMMMMMM] → 16 bits per weight
```

#### INT8 (8-bit Integer)
- Fixed-point representation, no floating decimal
- Quarter the memory of FP32
- Requires **quantization** — converting weights from floats to integers
- Slight accuracy loss, but much faster on supported hardware

```
Structure: [IIIIIIII] → 8 bits per weight
```

### Practical Memory Impact

For a **7B parameter model**:
| Precision | Memory Required | Example GPU |
|-----------|-----------------|-------------|
| FP32 | ~33.6 GB | Needs multiple high-end GPUs |
| FP16 | ~16.8 GB | A100 40GB, RTX 4090 |
| INT8 | ~8.4 GB | RTX 3090, RTX 4080 |

### When to Use Each Precision

| Scenario | Recommended Precision |
|----------|----------------------|
| Training from scratch | FP32 or mixed FP16/FP32 |
| Fine-tuning | FP16 or BF16 |
| Production inference | FP16 (best balance) |
| Edge/mobile deployment | INT8 or INT4 |
| Limited GPU memory | INT8 or quantized |

### Key Takeaways

1. **Lower precision = smaller memory footprint** — FP16 is half the size of FP32, INT8 is a quarter
2. **The 20% overhead** accounts for activation memory, KV cache, framework overhead, and CUDA context
3. **This is just inference** — training requires significantly more memory (gradients, optimizer states, etc.)
4. **Quantization trades accuracy for efficiency** — INT8 models may have slight quality degradation but run much faster

---

Returning to our GPT-Neo example from section 4.1, the model has 2.7 billion parameters. We loaded it in FP32, so applying the preceding formula gives us 12.96 GB. Loading the same model in FP16 reduces this to approximately 6.48 GB. If we run the same calculation for a different model, such as Mistral 7B in FP16, we get approximately 16.8 GB.
